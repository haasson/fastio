## Журнал действий
- **URL:** /audit-log
- **Пермишен:** audit_log.view (владелец видит всегда)

### Что это
Единая лента событий заведения: изменения конфигурации (меню, категории, модификаторы, добавки, комбо, теги, филиалы, столы, брони, услуги, промокоды, акции, роли, сотрудники, приглашения, настройки) + события заказов (создание, смена статуса). Записи создаются автоматически на уровне БД — изменить или удалить их нельзя (append-only).

### Действия
- Фильтр «Действие»: Создано / Изменено / Удалено / Восстановлено (мультивыбор)
- Фильтр «Объект»: сгруппированный мультивыбор по разделам; видны только разделы доступные тенанту (вертикаль + включённые модули)
- Поиск: по названию объекта и имени сотрудника
- Скоуп по филиалу: подчиняется выбору филиала в сайдбаре; конкретный филиал = его события + общие («Всё заведение»); в режиме «все филиалы» у филиальных записей чип с именем филиала
- Кнопка «Обновить»; лента листается бесконечно (старые записи подгружаются при прокрутке)
- Фильтры запоминаются в браузере

### Отображение
- Для изменений — дифф «что было → что стало» на русском; цены подсвечены (рост красным, снижение зелёным)
- Заказ за столом (dine-in) показан как объект-стол, обычный заказ — по номеру
- Автор: имя и роль сотрудника на момент действия; системные действия — «Система»
- Реордер списков (перетаскивание) не логируется

### Частые вопросы
- «Кто изменил цену/удалил блюдо?» → Журнал действий, фильтр по объекту «Меню» или поиск по названию
- «Почему запись нельзя удалить?» → журнал append-only, это защита истории изменений
