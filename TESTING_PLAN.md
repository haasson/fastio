## ⚠ Прод-контекст. Прочитай ДО начала

На проде **нет реальных клиентов** — это «холодный прод», песочница на проде. Поэтому почти всё можно: SQL-хаки в Supabase Dashboard, прямые DELETE тенантов после теста, фейковые payment-webhook payload'ы, force-suspended через UPDATE. Никого не задеваешь — играй смелее.

Но прод-**инфраструктура** реальна, поэтому всё-таки немного дисциплины:

1. **Реальный email-ящик.** Используй один свой с plus-алиасами: `you+t1@gmail.com`, `you+manager-t1@gmail.com` и т.д. `example.com` НЕ работает — почта не доставится, и регистрация повиснет.
2. **Префикс `TEST`** в именах тенантов (`TEST T1 Кафе Луна`) и slug (`test-t1-cafe`) — чтобы потом одной командой найти и удалить.
3. **Прод-Telegram-бот реальный.** Создавай свои тестовые группы, не флуди в production-чаты команды.
4. **Yandex Delivery / Dadata / Yandex Maps** — на проде используют ключи с лимитами. Не запускай нагрузочные тесты.
5. **Custom domain** — если тестируешь, используй свой тестовый домен, не чужие.
6. **payment-webhook** — даже с фейковыми payload'ами лимитируй число вызовов (10–20 хватит).
7. **Логи** живут в Supabase Dashboard → Edge Functions → Logs. При ошибках сохраняй request_id.
8. **Зачистка после теста** — раздел 9.5. Холодный прод сегодня не повод оставлять `TEST T1 Кафе Луна` навечно.

Всё остальное — **играй**: прямые UPDATE для биллинг-edge, DELETE тенантов через SQL, replay webhook с фейковой подписью, cross-tenant POST для проверки RLS. SQL Editor в Supabase Dashboard в полном распоряжении.

## 0. Как пользоваться этим планом

### 0.1 Почему «по тенантам», а не «по фичам»

Часть настроек выбирается **один раз и навсегда** (тип бизнеса, стиль меню, slug). Часть **переключается налету** (план, модули, темы). Тестировать всё на одном тенанте не получится — нужны разные «формы».

Поэтому план строится так:

1. **Реестр тенантов** (раздел 1) — таблица «кто за что отвечает».
2. **Pre-flight** (раздел 2) — что подготовить один раз перед началом.
3. **T1…T4** (разделы 3–6) — последовательное прохождение каждого тенанта.
4. **Упражнения поверх существующих тенантов** (раздел 7) — RBAC, биллинг-edge, безопасность, realtime — выполняется на уже созданных.
5. **Хитрые кейсы / data integrity / race** (раздел 8) — отдельная фаза «давай попробуем сломать».
6. **Финальный sweep** (раздел 9) — закрывающий чеклист.

### 0.2 Что залочено / что меняется

| Параметр | Залочено | Меняется налету |
|---|---|---|
| `businessType` (retail / services) | ✅ выбирается в онбординге, потом не меняется через UI | — |
| `menuStyle` (food / catalog, только для retail) | ✅ онбординг, шаг есть только для retail | — |
| `slug` | ✅ задаётся юзером **на лендинге при регистрации** (с realtime check уникальности), в админке UI смены нет | — |
| `customDomain` | — | ✅ да, через специальную фичу (Edge Function `add-custom-domain`) |
| `plan` (showcase / start / pro) | — | ✅ через `/account/billing` → `billing_change_plan` RPC. По умолчанию после регистрации — `start` с триалом 14 дней |
| Любой `modules.*` | — | ✅ через `/settings/modules` |
| Кол-во филиалов | — | ✅ при включённом модуле `branches` |

> Поэтому невозможно: «retail-тенант, который потом превратится в services». Для каждой вертикали — свой тенант.

### 0.3 Что реально задаёт лендинг и онбординг

**Регистрация на лендинге `fastio.ru` (секция «Попробуйте бесплатно», якорь `#try`)**:
- Поле «Название заведения» → `tenant.name`
- Поле «Адрес сайта» → `tenant.slug` (например `green-cafe`), с realtime-проверкой уникальности и `/api/check-slug`. Слаг становится поддоменом: `<slug>.fastio.ru`
- Поле «Email» → создаётся Auth-юзер через invite + создаётся тенант через `self_register_tenant` RPC
- Дефолтные значения: `plan = start`, `trialEndsAt = +14 дней` (из `billing_config.trial_days`), `businessType = null`, `menuStyle = 'food'`, `modules` пустые
- На почту приходит письмо со ссылкой → клик → `admin.fastio.ru/set-password` → пароль → автологин → попадаешь на онбординг-визард

**Онбординг-визард в админке (после установки пароля)**:

| Шаг | Retail (food) | Retail (catalog) | Services |
|---|---|---|---|
| 1. type | businessType | businessType | businessType |
| 2. menuStyle | food/catalog | food/catalog | **пропущено** |
| 3. plan | showcase/start/pro + auto-modules по плану | то же | то же |
| 4. info | name + phone + timezone | то же | то же |
| 5. branch | первый филиал (или opt-out) | то же | то же |
| 6. modules | **только delivery/pickup/dineIn** | **delivery/pickup** (dineIn скрыт на catalog) | **пропущено** |
| 7. complete | onboardingCompleted=true | то же | то же |

**Важно:**
- На шаге `type` в wizard'е юзер уже залогинен и в его тенанте, `name + slug + email` уже заполнены с лендинга. На шаге `info` `name` можно переписать (он pre-fill), а slug там не меняется.
- Прочие модули (`modifiers`, `addons`, `promotions`, `combos`, `kitchen`, `customers`, `customRoles`, `branches`, `services` для retail) **в онбординге не настраиваются**. Их включаешь после, в `/settings/modules`. Это нужно держать в голове на каждом T*. (Отдельного модуля `reservations` нет — онлайн-брони входят в `dineIn`.)

### 0.4 Условные обозначения

- `[ ]` — чекбокс
- **HP** = happy path; **EC** = edge case; **NEG** = негатив (ожидается ошибка); **CROSS** = межмодульный; **RT** = realtime
- **P0** = критично, **P1** = важно, **P2** = средне

### 0.6 Режим «агент рядом»

План рассчитан на прохождение вместе с агентом. У агента SSH-доступ к проду и полные права:
- SQL: `docker exec -i <container> psql -U postgres fastio`
- Supabase REST API с service-role ключом
- curl к admin/storefront API
- Чтение docker logs и Edge Function logs (Supabase Dashboard)

Маркеры в разделах:
- 🤖 **Агент готовит** — агент выполняет SQL/curl/RPC, говорит «готово» с конкретными ID
- 👤 **Ты** — что делать в браузере
- 🔍 **Агент проверяет** — SQL/лог-чек после твоих действий

Клиентов на проде нет — агент может делать всё без ограничений.

### 2.6 Edge Functions — секреты и webhook'и (deploy-чеклист)

Секреты задаются один раз через `supabase secrets set` (Cloud Dashboard → Edge Functions → Secrets). Если хотя бы один не выставлен — соответствующая функция вернёт 401/500, и фича отвалится молча (без UI-ошибки, только в Edge logs). Проверь до начала прогона:

- [x] **`FASTIO_INTERNAL_TOKEN`** — server-to-server секрет для `send-new-tenant-email`. **Без него welcome-письма после регистрации тенанта не приходят.**
  - `supabase secrets set FASTIO_INTERNAL_TOKEN=$(openssl rand -hex 32)`
  - Тот же токен прокинуть в env обоих caller'ов:
    - landing: `NUXT_FASTIO_INTERNAL_TOKEN=<значение>` (Vercel/Coolify env)
    - backoffice: `NUXT_FASTIO_INTERNAL_TOKEN=<значение>`
  - Проверка: после регистрации нового тенанта через лендинг — на email должно прийти «Новое заведение... добавлено». Если не пришло — Supabase Dashboard → Edge Functions → `send-new-tenant-email` → Logs, ищи 401 или «Server misconfigured».
- [x] **`SMTP_USER` / `SMTP_PASS`** — общие SMTP-креды (используют `send-new-tenant-email`, `send-recovery-email`, `invite-member`). Без них все письма уходят в `console.log` вместо отправки. Проверь хотя бы по одному из сценариев (сброс пароля проще всего: `/forgot-password`).
- [x] **`COOLIFY_TOKEN` / `COOLIFY_PROJECT_ID`** — для `add-custom-domain`. Нужны только если будешь тестировать подключение кастомных доменов. Кастомные домены управляются через Coolify API (`feat(add-custom-domain): Vercel API → Coolify API`).
- [x] Smoke: после ставки секретов вызови `supabase functions list` и убедись, что все функции `ACTIVE` (не `ERROR` при старте).

## 11. Если что-то пошло не так

- **Тенант не создаётся** → почта в спаме? Supabase Dashboard → Authentication → Users — есть ли запись? Если есть юзер, но нет тенанта в `tenants` — это баг (видимо `self_register_tenant` RPC упал) → SQL Editor посмотри логи функции, потом эскалируй с деталями
- **Slug при регистрации не проходит проверку** → `/api/check-slug` уже занят либо невалидные символы. Попробуй другой `test-*-X` slug
- **Realtime не работает** → DevTools → Network → WS — есть ли активное соединение к Supabase? Можно перелогиниться
- **Тогглы не сохраняются** → DevTools Console → ошибки RLS/permissions. Модуль может быть locked планом или `moduleToggleChecks` блокирует из-за активных зависимостей
- **Витрина не открывается по slug** → проверь DNS (`https://<slug>.fastio.ru` должен резолвиться через wildcard), tenant middleware, статус `subscription.status != 'suspended'`
- **Telegram не приходит** → Supabase Dashboard → Edge Functions → Logs (ищи `telegram-notify` или подобный). Vault-секреты `telegram_bot_token` / `telegram_internal_secret` должны быть выставлены
- **Reminders не шлются** → Supabase Dashboard → Database → Cron Jobs → `send-appointment-reminders` (раз в минуту) → проверь last_run/last_error. Vault-секреты `appointment_reminder_url` и `reminder_cron_secret`
- **`billing_change_plan` возвращает «Already on this plan»** → выбери другой план или вручную в SQL:
  ```sql
  UPDATE tenants SET subscription = jsonb_set(subscription, '{plan}', '"showcase"') WHERE slug = 'test-t4-vitrina';
  ```
- **Что-то застряло в неконсистентном состоянии** → можно `DELETE FROM tenants WHERE slug = '...'` и зарегистрировать заново. Клиентов нет — терять нечего.
