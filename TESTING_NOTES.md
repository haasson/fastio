- 🐛 ~~После включения предзаказа статус «Запланировано» не появлялся в дропдауне~~ **FIXED**: после `ensureScheduledHoldingStatus` вызываем `statusesStore.reload()` — Realtime-гонка устранена.

- 💡 **Слоты будущих дней начинаются с открытия без lead-time** — `pickupLeadMinutes`/`deliveryLeadMinutes` применяются только к сегодняшним слотам (отодвигают от текущего момента). Завтра — слоты с 07:00 (с открытия). На практике: самовывоз в 07:00 = ресторан только открылся, готовить некогда. Возможное решение: первый слот каждого дня = `open + leadMinutes`, не только сегодня. Либо отдельный параметр `openBufferMinutes`.
- 🐛 **Нет буфера закрытия для ASAP-заказов** — `closeBufferMinutes` живёт только в `orderSchedulingConfig` и используется исключительно для генерации слотов предзаказа. Для обычных (ASAP) заказов сервер делает только бинарную проверку `isOpenNow` без буфера. Сценарий: кафе до 23:00, в 22:59 прилетает заказ — принимается. Нужен отдельный параметр типа `stopAcceptingMinutesBefore` (или переиспользовать `closeBufferMinutes`), который применяется и к ASAP-заказам. Место: `order-delivery.ts` → `isOpenNow` → передавать буфер.
- 💡 **Буфер закрытия по типу (доставка/самовывоз) — доменная развилка, не правка UI.** Открытый буфер уже раздельный (`deliveryLeadMinutes`/`pickupLeadMinutes`), закрытый — один `closeBufferMinutes` на оба типа → непоследовательно. НО прежде чем делать второе поле, надо решить семантику «закрытия», оно перегружено на 3 момента: (1) закрытие для посетителей (двери / самовывоз не отдать), (2) выключение кухни (последнее блюдо приготовлено), (3) последняя доставка уехала/доехала.
  - Самовывоз: cutoff = `закрытие − готовка − запас «дойти»` (клиент должен физически прийти, пока открыто).
  - Доставка: **если разрешено доставлять после закрытия кафе** (приготовили до выключения кухни, курьер везёт позже) — реальный констрейнт «успеть приготовить», cutoff = `закрытие_кухни − готовка`, путь не упирается в закрытие → доставку можно принимать **дольше** самовывоза. Если политика «всё завершено до закрытия» — доставка отрубается **раньше** (нужен и путь). Т.е. кто больше — зависит от политики, один скаляр не выражает.
  - **Решить до реализации:** разрешена ли доставка после закрытия витрины? Нужна ли отдельная сущность «последний заказ на кухню» vs «закрытие витрины»? После этого: схема (`closeBufferDeliveryMinutes`/`closeBufferPickupMinutes` или модель «kitchen last-order»), слот-ген, серверный cutoff по типу — **в одном заходе с ASAP-багом выше**.
  - Side-effect: при 4 буферах (2 закрытия + 2 открытия) сетка станет 2-колоночной (~350px на поле), и длинный лейбл «Буфер при открытии — самовывоз» (сейчас чуть не влазит в 3-колоночной) влезет сам.
