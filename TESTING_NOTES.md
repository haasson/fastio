### 3.5 Заказы

- 💡 **История заказов клиента → открывать в новой вкладке** — при клике на прошлый заказ из истории клиента (витрина `/account/orders`) хочется `target="_blank"`, а не переход в той же вкладке.
- 🐛 **Нельзя редактировать адрес/состав принятого заказа** — `editItems` и `editAddress` заблокированы для `in_progress` в `useOrderStatus.ts`. Для `in_progress` доступны только: телефон клиента, оплата, сумма доставки. Реальные кейсы: клиент перезвонил → добавил блюдо / исправил адрес. Нужно разрешить с предупреждением (особенно если kitchen уже взяла в работу). Адрес: дополнительно перепроверять зону доставки после смены.
- 🐛 **Нельзя добавить позицию в принятый заказ** — `editItems: g === 'new'` в `useOrderStatus.ts`. Штатный сценарий: клиент перезвонил через минуту и попросил ещё бургер. Сейчас тенант не может добавить блюдо в заказ группы `in_progress`. Нужно: разрешить редактирование состава для `in_progress` + предупреждение если kitchen-модуль уже взял заказ в работу + audit-событие `items_updated` с diff + kitchen должна получить новые тикеты для добавленных позиций.

### 3.5.4 Предзаказ / Буфер закрытия

- 🐛 ~~После включения предзаказа статус «Запланировано» не появлялся в дропдауне~~ **FIXED**: после `ensureScheduledHoldingStatus` вызываем `statusesStore.reload()` — Realtime-гонка устранена.
- ✅ **Начальный статус предзаказа** — падает в «Новые» как любой заказ. Оператор подтверждает → переводит в «Запланировано» вручную (если ещё рано) → автопереход в `nextStatusId` когда время настало. Это единственный флоу, поведение корректное.
- 💡 **Слоты будущих дней начинаются с открытия без lead-time** — `pickupLeadMinutes`/`deliveryLeadMinutes` применяются только к сегодняшним слотам (отодвигают от текущего момента). Завтра — слоты с 07:00 (с открытия). На практике: самовывоз в 07:00 = ресторан только открылся, готовить некогда. Возможное решение: первый слот каждого дня = `open + leadMinutes`, не только сегодня. Либо отдельный параметр `openBufferMinutes`.
- 🐛 **Нет буфера закрытия для ASAP-заказов** — `closeBufferMinutes` живёт только в `orderSchedulingConfig` и используется исключительно для генерации слотов предзаказа. Для обычных (ASAP) заказов сервер делает только бинарную проверку `isOpenNow` без буфера. Сценарий: кафе до 23:00, в 22:59 прилетает заказ — принимается. Нужен отдельный параметр типа `stopAcceptingMinutesBefore` (или переиспользовать `closeBufferMinutes`), который применяется и к ASAP-заказам. Место: `order-delivery.ts` → `isOpenNow` → передавать буфер.

### Системная проблема: часы работы

- 🐛 **Часы работы — разорванная архитектура при включённых филиалах:**
  1. **UX**: в общих настройках написано «настраивай в филиалах», в филиале написано «по умолчанию — из общих». Нет ни одного места где можно просто взять и выставить часы.
  2. **Слоты предзаказа**: `useSchedulingSlots(tenant, ...)` в `checkout.vue` всегда берёт расписание тенанта, игнорируя расписание конкретного филиала (критично для pickup — клиент выбрал Парк, а слоты считаются по тенанту).
  3. **Валидация заказа**: `order-delivery.ts` — branch-schedule fallback на tenant-schedule работает правильно, но при pickup должен использоваться только schedule выбранного филиала.
  - Нужно решить дизайн: или tenant-schedule = мастер для всех филиалов (тогда UI должен это отражать), или каждый филиал полностью самостоятелен (тогда `useSchedulingSlots` должен принимать branch-schedule).
