<objective>
Add Lighthouse CI to the GitHub Actions pipeline so that every PR to `main` runs a real Lighthouse audit against the staging storefront with the mobile preset and BLOCKS the build if LCP ≥ 2.5s or CLS ≥ 0.1. Closes PERF-02.

Purpose: Plan 02 made the storefront fast in theory; without an automated guard a regression (an unoptimized image, a heavy third-party script, a layout shift) ships silently and the next ad-spend traffic burns on a slow page. LHCI is the last vertical-slice piece — after this plan ships, a developer who introduces a perf regression sees a red check on their PR and cannot merge until they fix it.

Output: `lighthouserc.json` at repo root with LCP < 2500ms and CLS < 0.1 as ERROR-level assertions on the mobile preset; a new `lhci` job in `.github/workflows/ci.yml` pinned to Node 20 + `@lhci/cli@0.14.0` + staging URL.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LHCI Chrome → staging storefront URL | LHCI takes screenshots + HAR traces of the audited page. If the URL is production, real customer data (orders, names, addresses) may end up in the temporary-public-storage HTML report. |
| `temporary-public-storage` LHCI upload | Public, ephemeral but indexable Google Storage bucket. Anyone with the report URL can view the screenshots. |
| GitHub Actions secrets → `pnpm build` | Build-time env vars are exposed to the running step. NUXT_PUBLIC_* are intentionally public anyway. Service-role keys MUST NEVER be exposed in this job. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-08 | Information disclosure | `ci.collect.url` in lighthouserc.json | mitigate | URL is `https://demo.fastio.ru/` (staging tenant), NOT prod. Lighthouse screenshots get uploaded to Google temporary-public-storage; staging contains seed data only. RESEARCH.md security_context flags this explicitly. Task 1 acceptance asserts no localhost / no prod URL. |
| T-4-09 | Information disclosure | LHCI service-role exposure in build step | mitigate | The `build storefront` step env block exposes ONLY `NUXT_PUBLIC_SUPABASE_URL` and `NUXT_PUBLIC_SUPABASE_ANON_KEY` — NOT `NUXT_SUPABASE_SERVICE_ROLE_KEY`. Storefront build never needs service-role; SEC-02 invariant. Task 2 acceptance asserts the env block excludes service-role. |
| T-4-10 | Tampering | `npx @lhci/cli@0.14.0` invocation | mitigate | Exact version pin (`@0.14.0`). RESEARCH.md Package Legitimacy Audit verified: Google Chrome OSS, github.com/GoogleChrome/lighthouse-ci, ~6.5y old, high downloads (Approved). Version selected for Node 20 compatibility (0.15.x requires Node >=22.19). Pin prevents accidental float to a version that breaks the runtime. |
| T-4-SC2 | Tampering | `npx` fetching @lhci/cli on every run | accept | npx caches the package per-runner; cold cache fetches from npm registry. Without lockfile pinning for npx packages, supply-chain compromise of @lhci/cli is the residual risk. Mitigation: exact version pin makes a malicious 0.14.0 republish detectable; alternative (`pnpm add @lhci/cli@0.14.0 -D`) was considered but not adopted because LHCI is CI-only and bloats the lockfile. |
</threat_model>
