<objective>
Закрыть E2E-03 vertical slice: создать новый чистый API-тест `tests/e2e/cross-tenant.spec.ts`, который проверяет, что валидная customer-сессия tenant-A не получает данные tenant-B через манипуляцию Host-хедером. Это контракт-тест на tenant isolation: tenantDb proxy инъектит `.eq('tenant_id', db.tenantId)`, плюс RLS на уровне Supabase — оба механизма должны давать 401/403/404 для cross-tenant запроса.

Назначение: E2E-03 — единственный success criterion фазы 3, связанный с безопасностью, и единственный API-тест (без UI). Storefront tenant middleware резолвит tenant из Host-хедера, а `tg_session` cookie привязана к конкретному tenant_id. Если код где-то теряет проверку — атакующий может с куки tenant-A читать данные tenant-B. Тест верифицирует это поведение и предотвращает регрессии при будущих рефакторах tenantDb или middleware.

Output: новый файл `tests/e2e/cross-tenant.spec.ts` (API-only, без UI); расширение `supabase/seed/e2e-staging.sql` — добавление второго тенанта `e2e-b` с фиксированным UUID, чтобы staging тест мог использовать реальный tenant-B (а не несуществующий slug, который вернёт 503 от middleware и не проверит реальную RLS-изоляцию).

    ```typescript
    import { test, expect } from '@playwright/test'
    import { fixtures } from './fixtures'

    // E2E-03: cross-tenant isolation. Один и тот же tg_session token, но разные
    // Host-хедера → middleware резолвит разные tenant_id → tenantDb proxy и RLS
    // должны заблокировать доступ к данным "чужого" тенанта.
    //
    // Это API-тест (без UI). Playwright `request` fixture поддерживает кастомный
    // Host хедер с Playwright 1.60+.

    // tenant-B slug определяется по среде:
    // - локально: используется services-start (второй тенант demo seed)
    // - staging: используется e2e-b (добавлен в 03-01 seed via 03-04 Task 1)
    const TENANT_B_SLUG = process.env.SUPABASE_STAGING_URL ? 'e2e-b' : fixtures.servicesTenantSlug
    const STOREFRONT_PORT = 4711

    test('cross-tenant isolation: tenant-A session is rejected for tenant-B API', async ({ request }) => {
      const tenantAHost = `${fixtures.retailTenantSlug}.localhost:${STOREFRONT_PORT}`
      const tenantBHost = `${TENANT_B_SLUG}.localhost:${STOREFRONT_PORT}`
      const cookie = `tg_session=${fixtures.sessionTokenRetail}`

      // Sanity: tenant-A с своей кукой — допустимый response (200 или 401 если
      // customer не зарегистрирован в этом env). НИКОГДА не 403, потому что 403
      // означает "auth есть, доступ запрещён" — это симметричный нарушитель.
      const responseA = await request.get(`http://localhost:${STOREFRONT_PORT}/api/customer/orders`, {
        headers: {
          'Host': tenantAHost,
          'Cookie': cookie,
        },
      })
      expect(responseA.status(), `tenant-A own session should not be 403, got ${responseA.status()}`).not.toBe(403)

      // ОСНОВНОЙ assertion: тот же токен с Host tenant-B → должно быть 401/403/404.
      // Если возвращается 200 → CRITICAL leak: проверка изоляции провалена.
      const responseB = await request.get(`http://localhost:${STOREFRONT_PORT}/api/customer/orders`, {
        headers: {
          'Host': tenantBHost,
          'Cookie': cookie,
        },
      })
      expect(
        [401, 403, 404],
        `cross-tenant leak: tenant-A session received status ${responseB.status()} for tenant-B API; expected 401/403/404`,
      ).toContain(responseB.status())

      // Дополнительная защита: если статус 200, тело НЕ должно содержать данные
      // tenant-A. (Эта ветка не должна выполниться — assertion выше упадёт раньше.)
      if (responseB.status() === 200) {
        const body = await responseB.json()
        // Это означает критический leak — упасть в любом случае.
        throw new Error(`CRITICAL: cross-tenant 200 response body: ${JSON.stringify(body).slice(0, 500)}`)
      }
    })
    ```
