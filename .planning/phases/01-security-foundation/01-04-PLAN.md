<objective>
Establish an isolated staging Supabase Cloud project, wire migrations to push to it automatically, create a minimal E2E seed, and rewrite the Playwright globalSetup so it can target staging via the service-role HTTP API (no docker dependency) while preserving the existing local-docker path unchanged.

Purpose: SEC-04 closes the "E2E tests must not share a database with anything else" gap. Today the E2E suite runs against a local Supabase via docker. To unblock Phase 3 (E2E flows that must run reliably in CI without depending on a developer's laptop being on), Phase 1 sets up the destination environment and the connectivity scaffolding. The actual rewiring of e2e-smoke.yml and e2e-nightly.yml to target staging happens in Phase 3 — Phase 1 only delivers the staging project, the migrate push, the seed file, and the dual-path globalSetup.

Output: a new optional step in migrate.yml that pushes applied migrations to staging when staging secrets are present; a new minimal seed at supabase/seed/e2e-staging.sql with fixed-UUID fixtures; a refactored tests/e2e/global-setup.mjs that branches between setupLocal (existing docker path, preserved verbatim) and setupStaging (new HTTP-API path); a human-verify checkpoint that walks the developer through Supabase Cloud project creation and GitHub Secrets configuration. This plan depends on Plan 01-03 (the migrate.yml RLS audit step must already exist before we touch the same file again).
</objective>
