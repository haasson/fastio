<objective>
Harden the storefront tenant middleware and getTenantDb so the system fails loudly and correctly on misconfiguration instead of silently degrading or leaking tenant existence information.

Purpose: SEC-03 closes three tenant-context failure modes — missing Host header, unknown tenant domain, and missing tenantId in request context — by surfacing the correct HTTP status code for each. Currently the unknown-tenant path returns 404 (which enables tenant enumeration and is wrong for load balancer retry semantics) and the missing-tenantId path returns 400 (which signals a client error instead of the server bug it actually is).

Output: tenant.ts adds an explicit Host guard and switches the unknown-tenant terminal throw from 404 to 503; tenantDb.ts switches the missing-tenant-context throw from 400 to 500; new unit tests cover the three behaviors; the existing tenantDb.test.ts case asserting 400 is updated to assert 500.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet → Nitro middleware | Untrusted Host header crosses here; attacker can spoof/omit/probe domain values |
| Nitro middleware → Postgres (service-role) | tenant lookup; failure to enforce context = service-role bypass on wrong tenant |
| Request handler → getTenantDb() | request context must contain a resolved tenantId; absence indicates a server bug |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | tenant.ts unknown-tenant path | mitigate | Replace 404 with 503 so attackers cannot enumerate valid vs invalid tenant domains via status code differentiation (D-06) |
| T-1-02 | Information Disclosure | tenant.ts missing-Host path | mitigate | Add explicit guard before any DB lookup; throw 503 with generic message so missing Host cannot be used to probe DB behavior (D-07) |
| T-1-03 | Elevation of Privilege | tenantDb.getTenantDb() | mitigate | Switch 400 → 500 so the absence of `event.context.tenantId` raises an alert in Sentry/GlitchTip (Phase 2) instead of looking like a client mistake — this catches middleware misconfiguration loudly (D-08) |
| T-1-04 | Denial of Service | Repeated bad-Host probes | accept | 503 is the correct retry-able signal; rate limiting is out of scope for Phase 1 and not regressed by these changes |
</threat_model>
