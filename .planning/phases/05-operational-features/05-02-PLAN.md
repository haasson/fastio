<!-- Executor does not need to explore — start from these locations. -->

Order status page: `apps/storefront/pages/order/[id].vue`
- Tenant-aware SSR route, no authentication required.
- IDOR guard: enforced server-side in `apps/storefront/server/api/orders/[id].get.ts` (one of: `?t=guest_token`, `tg_session` cookie, or `Authorization: Bearer` JWT).
- Polling: `setInterval(refresh, 15_000)` started on client mount, stopped once `statusGroup === 'completed' || 'cancelled'` (terminal state), or in `onUnmounted`.
- States rendered:
  - `pending && !order` → loading spinner ("Загружаем заказ...")
  - `!pending && !order` → 404 empty state ("Заказ не найден")
  - `order` → full order card with header, items list, totals, delivery info, "На главную" CTA.

Checkout flow: `apps/storefront/pages/checkout.vue`
- After successful order creation, navigates to `/order/${id}?t=${token}` where `token` is the `guest_token` returned from `POST /api/orders`.

Order creation: `apps/storefront/server/api/orders.post.ts`
- Generates a `guest_token` (UUID) at order creation and stores it in `orders.guest_token` for IDOR.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet → `/order/{id}` page | Anyone can request this URL; IDOR guard in server endpoint is the only protection |
| Public internet → `/api/orders/{id}` endpoint | Three accepted credentials: `?t=guest_token` (URL), `tg_session` cookie (Telegram login), or `Authorization: Bearer` JWT (Supabase session) |
| Storefront client (after polling) → `/api/orders/{id}` | Each 15s poll re-sends whichever credential the page started with |
| Verification artifact → `.planning/` | Documents what was tested; itself does not modify code or grant access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Information Disclosure | IDOR — guessing an order UUID | mitigate (already implemented) | `apps/storefront/server/api/orders/[id].get.ts` requires one of three valid credentials and returns 404 on failure (not 403 — avoids order-existence enumeration). Task 1 cites this; Task 2 step 6 actively probes it. |
| T-05-02-02 | Spoofing | Forged `?t=` token | mitigate (already implemented) | `guest_token` is a server-generated UUID stored in `orders.guest_token` at order creation; the server endpoint matches it via equality, not pattern. UUID space (2^122 effective) makes brute force infeasible for v1. |
| T-05-02-03 | Information Disclosure | Polling traffic leaks order existence | accept | The polling endpoint returns the order payload at the same rate to the same authenticated client; no oracle leak introduced by polling vs single fetch. Documented in this plan for verifier awareness. |
| T-05-02-04 | DoS | Open polling loop on tab kept open indefinitely | accept | `setInterval` stops in `onUnmounted` and on terminal status. A malicious client could keep the tab open and continue polling at 1 req / 15s — acceptable for current scale; rate limiting is a v2 concern. |
| T-05-02-05 | Tampering / Verification fraud | Verifier marks PASS without actually running steps 1–6 | mitigate | Task 2 acceptance language explicitly disallows marking PASS based on static citations alone; Task 1 leaves `[PENDING — Task 2 checkpoint]` placeholders that MUST be replaced with concrete observations before commit. |
| T-05-02-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan. Verified by RESEARCH.md Package Legitimacy Audit ("No external packages are being installed in this phase"). |
