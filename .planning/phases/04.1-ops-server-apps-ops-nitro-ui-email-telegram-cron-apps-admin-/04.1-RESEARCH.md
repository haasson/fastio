# Phase 04.1: Ops Server — Research

**Researched:** 2026-05-22
**Domain:** Nuxt/Nitro server-only app, Telegram notifications, pg_cron/pg_net, email templates, Coolify deployment
**Confidence:** HIGH

---

## Summary

Phase 04.1 extracts all "infrastructure HTTP endpoints" from `apps/admin` into a dedicated `apps/ops/` Nuxt/Nitro service with no frontend UI. The primary drivers are separation of concerns (admin should not be a platform infrastructure component) and the ability to deploy, scale, and restart ops independently.

The scope is larger than it looks. There are **10 Nitro handlers** in `apps/admin/server/api/telegram/`, but only some of them belong in ops. The split is:
- **Move to ops:** the 6 internal handlers called by the DB (pg_net triggers + pg_cron). These never need admin session context.
- **Stay in admin:** the 4 Telegram bot webhook handlers (`webhook.post.ts`, `auth-webhook.post.ts`, `glitchtip-alert.post.ts`, and `notify-alert.post.ts`) — the first two because the Vercel relay currently hardcodes `admin.fastio.ru` as target; the last two because `glitchtip-alert` is registered as a GlitchTip webhook alert destination and `notify-alert` is registered as a vault secret URL both pointing at admin.

Email templates (`supabase/templates/`) must be served as static HTML from ops because GoTrue reads them from an HTTP URL (configured via `GOTRUE_MAILER_TEMPLATES_*` env vars on the self-hosted GoTrue container). The 3 template files move to `apps/ops/public/email-templates/`.

The migration touches **vault.secrets** (7 URL records need updating), **Supabase Coolify env vars** (GOTRUE_MAILER_TEMPLATES_*), and the **Vercel relay** (TARGET URL in 2 files) after ops goes live.

**Primary recommendation:** Create `apps/ops/` as a minimal Nuxt app with `ssr: true`, no Vue UI, no auto-imports, just a Nitro server layer. Copy and adapt server utils from admin. Keep the Vercel relay pointing at admin for now — the webhook handlers (`webhook.post.ts`, `auth-webhook.post.ts`) are NOT moving in this phase.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| New order notification (pg_net trigger → HTTP) | ops server | — | Pure server-to-server, no admin session needed |
| New reservation notification (pg_net trigger) | ops server | — | Same as above |
| Appointment group notification (pg_net trigger) | ops server | — | Same as above |
| Table call notification (pg_net trigger) | ops server | — | Same as above |
| Appointment reminders (pg_cron every minute) | ops server | — | Cron trigger, no admin session needed |
| Edge error monitoring alert (pg_cron every 15m) | ops server | — | Monitoring infra, separate concern from admin UI |
| Telegram bot webhook (incoming from Telegram via relay) | admin server | — | Relay hardcodes admin.fastio.ru; moving requires Vercel relay config change — defer |
| Telegram auth webhook (incoming from Telegram via relay) | admin server | — | Same relay issue |
| GlitchTip webhook alert | admin server | — | GlitchTip alert URL configured externally, updating is manual ops work — defer or move together |
| Email templates static files | ops server (public/) | — | GoTrue fetches via HTTP URL; static public/ is simplest approach |
| Health endpoint | ops server | — | Coolify liveness probe, ops-specific |
| tel:// redirect endpoint | admin server | — | Called from Telegram inline buttons with adminUrl hardcoded in notify handlers; moving requires notify handler update first |

| `telegram_notify_appointment_group_url` | (derived from base or explicit) | `https://ops.fastio.ru/api/telegram/notify-appointment-group` |

Self-hosted GoTrue reads templates from HTTP URLs (not local files). The `GOTRUE_MAILER_TEMPLATES_*` env vars must be set in the Supabase Coolify service configuration. These are "hardcoded env" in Coolify (must be changed via Edit Compose File, not regular env vars).

| GoTrue env var | Template type | New value |
|---|---|---|
| `GOTRUE_MAILER_TEMPLATES_INVITE` | Invite email | `https://ops.fastio.ru/email-templates/invite.html` |
| `GOTRUE_MAILER_TEMPLATES_RECOVERY` | Password recovery | `https://ops.fastio.ru/email-templates/recovery.html` |
| `GOTRUE_MAILER_TEMPLATES_CONFIRMATION` | Email confirmation | `https://ops.fastio.ru/email-templates/confirm_signup.html` |

**Important:** These env vars may currently be unset (GoTrue falls back to built-in defaults). Setting them for the first time will activate the custom templates. The local dev `supabase/config.toml` already has `content_path` configured and is unaffected.

## Common Pitfalls

### Pitfall 1: The fallback regex in pg_net triggers

**What goes wrong:** Vault secrets `telegram_notify_appointment_group_url` and `telegram_notify_table_call_url` may not exist in production (they were optional, with fallback via `regexp_replace`). The fallback uses `'telegram_notify_url'` and does `regexp_replace(base, '/notify$', '/notify-...')`. If `telegram_notify_url` is updated to `ops.fastio.ru/api/telegram/notify` but the derived secrets were never created explicitly, the fallback still works. However, if those dedicated secrets DO exist but still point at admin, they take priority and notifications break.

**How to avoid:** When updating vault, update ALL 6 URL secrets. Don't rely on the fallback.

**Warning signs:** Notifications for appointment groups or table calls stop working after vault update but order and reservation notifications work fine.

### Pitfall 2: notify-alert.post.ts uses `@sentry/nuxt` directly

**What goes wrong:** The handler does `import * as Sentry from '@sentry/nuxt'` and calls `Sentry.captureMessage(...)`. If `@sentry/nuxt` is in the wrong module or the Sentry DSN is not configured in ops, this silently fails or throws.

**How to avoid:** Replace `Sentry.captureMessage(...)` with `reportError(new Error(...))` from `@fastio/shared/observability` — same pattern as all other handlers. Remove the direct `@sentry/nuxt` import from the ops copy of this file.

### Pitfall 3: telegram-bot-setup.ts plugin must NOT move to ops

**What goes wrong:** If the plugin is added to ops, it will try to `setWebhook` for both bots pointing at ops.fastio.ru. But the actual webhook handlers (`webhook.post.ts`, `auth-webhook.post.ts`) are in admin. This breaks Telegram authentication and tenant bot linking.

**How to avoid:** The plugin stays in admin. Ops has no `server/plugins/telegram-bot-setup.ts`. Verify this: after ops deploys, the Telegram webhook `getWebhookInfo` should still report `admin.fastio.ru` URLs.

### Pitfall 4: Deployment order — ops must be healthy before vault update

**What goes wrong:** If vault secrets are updated to `ops.fastio.ru` URLs before ops is deployed and the health endpoint returns 200, pg_net calls start hitting a non-existent server. During the window between vault update and ops being live, all Telegram notifications fail silently.

**How to avoid:** Deployment order must be: (1) deploy ops and verify `/api/health` returns 200, (2) update vault secrets, (3) verify a test notification works, (4) delete handlers from admin.

### Pitfall 5: Email templates served before GoTrue env vars are updated

**What goes wrong:** If GoTrue env vars (`GOTRUE_MAILER_TEMPLATES_*`) are set to `ops.fastio.ru` but ops is not yet deployed, all transactional emails (invite, recovery, confirmation) fall back to GoTrue built-in defaults. This is silent — no error, just different email layout.

**How to avoid:** Deploy ops first, verify `/email-templates/invite.html` returns 200, then update GoTrue env vars in Coolify and restart the Supabase service.

### Pitfall 6: Nuxt app.vue is required for ssr:true

**What goes wrong:** Without an `app.vue`, Nuxt may error during build or `nuxt dev` in SSR mode.

**How to avoid:** Create a minimal `app.vue` in `apps/ops/`:

```vue
<template>
  <div />
</template>

<script setup lang="ts">
// ops server — no frontend UI
</script>
```

Alternatively, use `ssr: false` — Nitro server still runs for API routes regardless of SSR mode. The `ssr: false` approach means no SSR initialization overhead and no need for `app.vue`. This is valid since ops never renders HTML.

### Pitfall 7: pnpm workspace — no Vue version drift

**What goes wrong:** Adding `vue` to `apps/ops/package.json` with a different version specifier than other apps causes two Vue instances in the monorepo. The `pnpm.overrides` in root `package.json` pins all `@vue/*` to `3.5.34` — the `vue` dep in `package.json` must use `^3.5.13` to match other apps, and the override handles the actual version.

**How to avoid:** Use the exact same specifier (`"vue": "^3.5.13"`) as `apps/admin/package.json`. The `pnpm.overrides` in root will pin it.

### Pitfall 8: Feature manifest validation hook only runs for admin

The pre-commit hook (`/.husky/pre-commit`) only runs `pnpm --filter admin exec lint-staged` and `pnpm --filter admin typecheck`. No `features:validate` for ops since ops has no `features/` directory. The hook does NOT need changes, but if you add an ops-specific lint script, wire it via `--filter @fastio/ops`.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes — server-to-server | `requireInternalSecret()`, `requireRelaySecret()` |
| V3 Session Management | no | No user sessions in ops |
| V4 Access Control | yes | All endpoints require auth; no public endpoints except /api/health and /email-templates/ |
| V5 Input Validation | yes | readBody() with typed shapes; orderId/tenantId presence checks |
| V6 Cryptography | no | Secrets compared via string equality; no encryption needed |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unauthorized trigger of notify endpoints | Spoofing | `requireInternalSecret()` — 500 if env missing, 403 on mismatch |
| Brute-force of appointment reminders cron | DoS on Telegram API | `requireReminderCronSecret()` — only pg_cron knows the secret |
| Vault URL hijacking (telegram_alert_url → attacker-controlled) | Tampering | `telegramAlertChatId` stays in env (not vault) — attacker can't redirect alert target even if vault URL is compromised |
| Bot token exposure in logs | Info Disclosure | Token only logged as URL prefix in `telegramFetch.ts` console.warn (masked by `[telegramFetch]` prefix) |

## Open Questions (RESOLVED)

1. **Should `glitchtip-alert.post.ts` move to ops?**
   - What we know: GlitchTip Slack-compatible webhook URL is configured in GlitchTip UI. The handler uses ops-bot-token + alert-chat-id. Architecturally it belongs in ops.
   - What's unclear: Is updating the GlitchTip webhook URL in scope? It requires a manual step in GlitchTip settings.
   - **RESOLVED: Deferred.** Риск обновления внешнего вебхука в GlitchTip UI добавляет blast-radius без значимого выигрыша в этой фазе. Хендлер остаётся в admin; записан в TECHDEBT.md (Plan 03-03 Task 5) с конкретным планом переноса.

2. **Should the Vercel relay update (pointing at ops for notify handlers) be in scope?**
   - What we know: The relay only forwards incoming Telegram webhook updates. The notify handlers (notify.post.ts etc.) are called by pg_net, NOT by the relay. The relay is irrelevant to the notify handler migration.
   - **RESOLVED: Out of scope.** Relay обрабатывает входящие Telegram webhook updates, а не исходящие pg_net notify-вызовы. Relay остаётся без изменений.

3. **Should Sentry/GlitchTip instrumentation be set up for ops in this phase?**
   - What we know: All other apps have `@sentry/nuxt` module wired.
   - What's unclear: Whether to create a separate GlitchTip project for ops or reuse the admin project.
   - **RESOLVED: Deferred.** `@sentry/nuxt` как Nuxt-модуль (с source-map upload) откладывается — нет отдельного GlitchTip проекта для ops. Зависимость `@sentry/nuxt` добавляется в package.json для `reportError` импорта из `@fastio/shared/observability`, но модуль не подключается в nuxt.config.ts.
