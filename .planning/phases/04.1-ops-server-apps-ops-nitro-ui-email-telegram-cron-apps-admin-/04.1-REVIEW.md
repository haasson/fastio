# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-23T00:00:00Z
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

This review covers the `apps/ops` Nitro server — a Nuxt 3 service handling Telegram
notifications and appointment reminders, migrated from `apps/admin`. Auth infrastructure
(`requireInternalSecret`, `requireRelaySecret`) is solid and correctly secured. The
`broadcastToTenantTelegram` utility is well-structured with dead-subscription pruning.

Two critical bugs were found: a systematic HTML injection vulnerability across two
notification handlers (user-controlled strings interpolated directly into Telegram's
`parse_mode: 'HTML'` messages), and a replyMarkup that unconditionally builds a "call"
button even when a phone number is unavailable. Four warnings cover silent Supabase error
discarding, token whitespace inconsistency, a Supabase relation filter alias mismatch, and
the `ssr: false` setting for a server-only app. Two info items flag minor quality issues.

## Critical Issues

### CR-01: HTML injection in `notify.post.ts` — user data unescaped in HTML parse mode

**File:** `apps/ops/server/api/telegram/notify.post.ts:58,61,64,67,78,80,81`

**Issue:** The handler sends Telegram messages with `parse_mode: 'HTML'` but injects
user-controlled DB values directly into the HTML string without escaping. Affected fields:

- `i.dish_name` — interpolated inside `<b>` tags (line 58)
- `i.modifiers[].optionName` — interpolated inside `<i>` tags (line 61)
- `i.addons[].name` — interpolated without escaping (line 64)
- `i.removed_ingredients[]` — interpolated without escaping (line 67)
- `order.customer_name` — interpolated without escaping (line 78)
- `order.address` — interpolated without escaping (line 80)
- `order.comment` — interpolated without escaping (line 81)
- `order.table_name` — interpolated without escaping (line 46)

A customer or operator could store `</b><b>` or `<a href="https://evil.com">click</a>` in
any of these fields. Telegram will parse injected tags, corrupting message formatting or,
in more sophisticated attacks, injecting inline links visible to all staff receiving the
notification. `notify-appointment-group.post.ts` and `notify-table-call.post.ts` correctly
use an `escapeHtml()` helper — this handler does not.

The same flaw exists in `notify-reservation.post.ts` at lines 47, 51, 52 for
`guest_name`, `table_name`, and `comment`.

**Fix:** Add the same `escapeHtml` helper used in `notify-appointment-group.post.ts` and
apply it to every user-supplied string before interpolation:

```typescript
function escapeHtml(s: string): string {
  return s
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
}

// notify.post.ts — apply at every interpolation point:
let line = `• <b>${escapeHtml(i.dish_name)}</b> × ${i.quantity} — ${formatPrice(itemTotal)}`
if (i.modifiers?.length) {
  line += `\n  <i>${i.modifiers.map((m) => escapeHtml(m.optionName)).join(', ')}</i>`
}
if (i.addons?.length) {
  line += `\n  + ${i.addons.map((a) => escapeHtml(a.name) + (a.quantity > 1 ? ` × ${a.quantity}` : '')).join(', ')}`
}
if (i.removed_ingredients?.length) {
  line += `\n  − ${i.removed_ingredients.map(escapeHtml).join(', ')}`
}
if (order.customer_name) text += `👤 ${escapeHtml(order.customer_name)}\n`
if (order.address) text += `📍 ${escapeHtml(order.address)}\n`
if (order.comment) text += `💬 ${escapeHtml(order.comment)}\n`

// notify-reservation.post.ts:
text += `👤 ${escapeHtml(reservation.guest_name)}\n`
if (reservation.table_name) text += `🪑 ${escapeHtml(reservation.table_name)}\n`
if (reservation.comment) text += `💬 ${escapeHtml(reservation.comment)}\n`
```

### CR-02: `notify-appointment-group.post.ts` — "call" button built without validating `phoneDigits`

**File:** `apps/ops/server/api/telegram/notify-appointment-group.post.ts:95-100`

**Issue:** The inline keyboard "Позвонить" button is constructed whenever `adminUrl` is
set, without checking that `phoneDigits` is a valid non-empty string:

```typescript
const phoneDigits = normalizePhone(group.customer_phone)  // line 95
const replyMarkup = adminUrl                               // line 96 — no phoneDigits check
  ? {
      inline_keyboard: [[{ text: '📞 Позвонить', url: `${adminUrl}/api/tel/${phoneDigits}` }]],
    }
  : null
```

`normalizePhone` strips non-digit characters and normalises the prefix. If
`group.customer_phone` is an empty string (which `NOT NULL` does not prevent at the DB
level), `normalizePhone('')` returns `''`, producing a URL like
`https://admin.fastio.ru/api/tel/` — a broken button delivered to every staff member
subscribed to the tenant's Telegram channel.

The other two handlers guard correctly: `notify.post.ts` guards with
`phoneDigits && adminUrl`, and `notify-reservation.post.ts` guards with
`adminUrl && phoneDigits`.

**Fix:**

```typescript
const phoneDigits = normalizePhone(group.customer_phone)
const replyMarkup = (adminUrl && phoneDigits)
  ? {
      inline_keyboard: [[{ text: '📞 Позвонить', url: `${adminUrl}/api/tel/${phoneDigits}` }]],
    }
  : null
```

## Warnings

### WR-01: Supabase errors silently discarded in `notify.post.ts` and `notify-reservation.post.ts`

**File:** `apps/ops/server/api/telegram/notify.post.ts:33` / `apps/ops/server/api/telegram/notify-reservation.post.ts:23`

**Issue:** Both handlers discard Supabase errors from their primary queries:

```typescript
// notify.post.ts
const { data: order } = await supabase.from('orders').select(...).single()
// error is never destructured — network failures, RLS policy mismatches,
// or Supabase outages cause order === null and silently return { ok: true }

// notify-reservation.post.ts
const [{ data: tenant }, { data: reservation }] = await Promise.all([...])
// errors from both queries are dropped
```

In contrast, `notify-appointment-group.post.ts` (lines 47-48), `notify-table-call.post.ts`
(line 36), and `send-appointment-reminders.post.ts` (line 67) all capture and `reportError`
their Supabase errors. When a Supabase query silently fails here, the notification is
dropped with no observability — a new order arrives and staff never knows.

**Fix:** Destructure and report errors:

```typescript
// notify.post.ts
const { data: order, error: orderErr } = await supabase.from('orders').select(...).single()
if (orderErr) reportError(orderErr, { ctx: 'telegram-notify.order', orderId, tenantId })
if (!order) return { ok: true }

// notify-reservation.post.ts
const [{ data: tenant, error: tenantErr }, { data: reservation, error: reservationErr }] = ...
if (tenantErr) reportError(tenantErr, { ctx: 'telegram-notify-reservation.tenant', tenantId })
if (reservationErr) reportError(reservationErr, { ctx: 'telegram-notify-reservation.reservation', reservationId })
```

### WR-02: `telegramTenantBotToken` not trimmed in three handlers

**File:** `apps/ops/server/api/telegram/notify-reservation.post.ts:16` / `notify-appointment-group.post.ts:25` / `notify-table-call.post.ts:20`

**Issue:** `notify.post.ts` reads the token as `config.telegramTenantBotToken?.trim()` (line
26), but the three other handlers that use the same token do not trim it:

```typescript
// notify-reservation.post.ts line 16
const token = config.telegramTenantBotToken        // no trim

// notify-appointment-group.post.ts line 25
const token = config.telegramTenantBotToken        // no trim

// notify-table-call.post.ts line 20
const token = config.telegramTenantBotToken        // no trim
```

A leading/trailing space in the env var (copy-paste accident in Coolify) would cause all
three handlers to silently build a malformed Telegram API URL. The `!token` guard only
catches an empty string, not a whitespace-only one, because `' '` is truthy.

**Fix:** Apply consistent trimming in all three handlers:

```typescript
const token = config.telegramTenantBotToken?.trim()
```

### WR-03: Supabase relation filter uses table name instead of alias in `send-appointment-reminders.post.ts`

**File:** `apps/ops/server/api/telegram/send-appointment-reminders.post.ts:64-65`

**Issue:** The query selects the joined resource with an alias:

```typescript
appointment:appointments!inner (
  starts_at,
  service_name,
  customer_name
)
```

The alias is `appointment` (singular). The subsequent filters use the table name:

```typescript
.gte('appointments.starts_at', cutoff)   // line 64 — 'appointments' not 'appointment'
.lte('appointments.starts_at', horizon)  // line 65
```

PostgREST documentation states that when a resource alias is defined, the filter key
should use the alias name, not the original table name. Using the wrong name may cause
PostgREST to silently ignore the filter, meaning the query fetches **all unsent reminders**
(no time-window constraint) instead of only the relevant time window. Correctness is
preserved by the in-process JS filter at line 73-76, but the DB query becomes unbounded —
on a busy multi-tenant deployment with thousands of scheduled reminders, this could
materially increase memory and network overhead every cron tick.

**Fix:**

```typescript
.gte('appointment.starts_at', cutoff)
.lte('appointment.starts_at', horizon)
```

### WR-04: `ssr: false` in `nuxt.config.ts` for a server-only API application

**File:** `apps/ops/nuxt.config.ts:6`

**Issue:** `ssr: false` disables server-side rendering and configures Nuxt to build a
client-side SPA shell. For an app that has no pages (only `server/api/` routes), this
setting is misleading and adds unnecessary build overhead (Vite client bundle). Nitro
server routes work regardless of the `ssr` setting, so functionality is unaffected, but
any future developer adding a page to `apps/ops` would get unexpected SPA behaviour.

The correct configuration for a pure API/Nitro server is either to keep `ssr: true`
(default) or to add `nitro: { preset: 'node-server' }` and leave SSR enabled, since no
Vue pages exist.

**Fix:**

```typescript
// Remove ssr: false or set explicitly to true:
export default defineNuxtConfig({
  compatibilityDate: '2024-11-01',
  devServer: { port: 4714 },
  // ssr: true is the default — remove the ssr: false line
  imports: { autoImport: false },
  components: false,
  runtimeConfig: { ... },
})
```

## Info

### IN-01: `resetServerSupabase` is exported but never called

**File:** `apps/ops/server/utils/supabase.ts:34-36`

**Issue:** `resetServerSupabase()` is a documented recovery mechanism for Supabase key
rotation (see PREPROD-120 comment). However, there is no call site anywhere in `apps/ops`
that actually invokes it. If a Supabase 401 is encountered (e.g., after key rotation), the
stale cached client will continue to be returned forever, requiring a full process restart.
The function is inert dead code in its current state.

**Fix:** Either wire `resetServerSupabase()` into an error-handling path (e.g., call it in
`telegramBroadcast.ts` when a Supabase `.error` with status 401 is detected), or remove it
and document that a process restart is the required recovery procedure for key rotation.

### IN-02: `BATCH_SIZE` and `BATCH_DELAY_MS` constants declared after their use site

**File:** `apps/ops/server/api/telegram/send-appointment-reminders.post.ts:87,100-101`

**Issue:** `BATCH_SIZE` and `BATCH_DELAY_MS` are used inside the event handler at lines
87-93 but declared at lines 100-101 (after the handler). `const` declarations are not
hoisted (temporal dead zone), so if any code path ever accessed these before module
initialisation completed, it would throw a `ReferenceError`. In practice this is safe
because the event handler is called at request time (after full module initialisation), but
placing constants after the code that depends on them is confusing and lints against the
project's top-of-file constant convention.

**Fix:** Move the constant declarations to the top of the file, before `defineEventHandler`:

```typescript
const BATCH_SIZE = 20
const BATCH_DELAY_MS = 1000

export default defineEventHandler(async (event) => {
  // ...
})
```
